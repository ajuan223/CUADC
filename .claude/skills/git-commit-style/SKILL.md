# Git Commit 风格规范

本 Skill 用于这个仓库中的 git commit 撰写与提交执行。目标不是“随便写一个能过的 commit message”，而是从**当前实际改动**和**项目既有历史**中提炼出一条符合你习惯、也符合仓库语义的提交信息。

## 这个仓库当前已经形成的风格

从近期历史可见，当前主风格不是单一 Conventional Commits，而是“**前缀 + 中文主题**”的实用混合风格：

- 常见前缀：`feat:` / `fix:` / `docs:` / `chore:`
- 某些历史提交也出现：`feat(web):`、`(fix)`、`(feature)`、`重构:`、`清理:`
- 主题主体多数为**中文**，直接描述本次变更解决的问题或补齐的能力
- 允许在一条 commit 里同时覆盖“功能 + 文档同步”，但前提是这些改动服务于同一件事

近期样例（作为风格参考，不是模板硬编码）：

- `feat: 新增飞后回放链路并同步 Skill 文档`
- `feat(web): 完善场地编辑器飞行路径预览及扫描生成算法`
- `docs: 同步 Skill 文档与代码现状，归档 refresh-skill-docs 变更`
- `chore: 重命名 skill 注册表入口`
- `(fix)修复扫描边界越界、起飞方向、巡航速度、投弹点降级策略，SITL全链路通过`

## 默认提交风格

### 1. 优先使用这些前缀

按意图优先选择：

- `feat:` —— 新增能力、扩展主链路、补齐关键功能
- `fix:` —— 修复明确 bug、回归、错误行为或错误产物
- `docs:` —— 纯文档、Skill、Spec、说明同步
- `chore:` —— 重命名、整理、脚手架、注册表、非业务行为调整

### 2. 中文主题必须写“结果”而不是流水账

优先写：

- 改动带来的能力
- 修复的症状
- 为什么这些文件被一起提交是合理的

避免写成：

- “修改若干文件”
- “更新代码”
- “处理一些问题”
- 纯文件名罗列

### 3. 标题长度与结构

- 优先单行标题，简洁直接
- 推荐格式：`<type>: <中文主题>`
- 除非明显更清晰，否则不要同时塞入过多顿号并列项
- 如果改动跨多处，但围绕同一目标，可写一个统一结果

好例子：

- `feat: 打通 flight_log 飞后回放链路`
- `fix: 修正扫描阶段边界越界与投弹点降级逻辑`
- `docs: 同步飞后回放相关 Skill 与注册表说明`

### 4. 何时写正文

下列情况应写 1~2 句中文正文：

- 一次 commit 同时包含代码、测试、Spec、Skill 文档
- 需要解释“为什么这些改动应该放在一起”
- 标题不足以表达边界，例如兼容性、回放字段约束、日志结构变化

正文写法要求：

- 第一段写为什么
- 第二段写这次实际收拢了什么
- 不写空话，不重复 diff

## message 决策规则

### 选择 `feat:`

满足任一条件优先用 `feat:`：

- 用户现在能做一件之前不能做的事
- 主链路能力被补齐（如 replay、SITL、投弹、场地编辑主流程）
- 虽然附带改了 docs/tests/spec，但核心是新增功能

### 选择 `fix:`

满足任一条件优先用 `fix:`：

- 修的是错误行为，而不是单纯补能力
- 目标是恢复正确结果、修复偏移、越界、时序、状态推进问题
- 测试新增只是为了锁住 bug

### 选择 `docs:`

仅当以下条件成立时使用：

- 没有业务代码变化
- 主要改的是 Skill、Spec、README、注册表、流程文档

### 选择 `chore:`

仅当以下条件成立时使用：

- 改动不改变业务语义
- 更像整理、重命名、迁移、注册入口调整、辅助脚本维护

## staging 规则

- 必须按文件名精确 `git add -- <file>`，不要用 `git add .` 或 `git add -A`
- 提交前必须检查是否混入无关文件
- 对于 `.env`、密钥、凭据、运行产物，一律排除
- 如果存在与当前目标无关的改动，先暂停并提醒用户决定是否拆 commit

## 提交前检查

提交前至少做三件事：

1. 看 `git status --short`
2. 看 `git diff --staged` 与 `git diff`
3. 看最近若干条 `git log --oneline`

如果改动涉及代码，应尽量补一句验证依据：

- 单元测试通过
- 前端逻辑测试通过
- 手工浏览器验证通过
- SITL 某轮通过

这些验证结论通常放在回复里，不一定写进 commit message；只有在 message 很需要解释边界时才写入正文。

## push 规则

- 只有用户明确要求时才 push
- 默认推到当前分支的 upstream，例如 `origin/<branch>`
- push 前确认当前分支与 upstream
- 除非用户明确要求，否则不要 force push

## 推荐执行流程

1. 检查 `git status --short`
2. 检查 `git diff --staged` / `git diff`
3. 检查 `git log --oneline -10`
4. 归纳改动属于 `feat/fix/docs/chore` 哪一类
5. 起草**中文**标题；必要时补 1~2 句中文正文
6. 精确 `git add -- <files>`
7. 用 HEREDOC 提交
8. 如用户要求，推送到 upstream 或指定 remote

## 输出要求

如果只是帮用户起草 message，返回：

- 推荐 commit 标题
- 可选正文
- 为什么这样归类（1 句）

如果用户要求你直接提交，执行完后返回：

- commit hash
- commit 标题
- 是否已 push，推到了哪里

## 禁止模式

- **禁止**脱离实际 diff 凭空写提交信息
- **禁止**把无关改动硬塞进一个 commit
- **禁止**英文标题配中文正文这种混乱风格，除非仓库已有明确要求
- **禁止**使用 `git add .` / `git add -A`
- **禁止**在未获明确要求时 push
- **禁止**为掩盖不清晰范围而写含糊标题，如“更新代码”“同步修改”“处理问题”
