## Context

Field Editor 当前只覆盖 `field.json` 的规划与预览，地图叠加能力主要服务于场地编辑；Striker 侧则已经具备 `flight_log` 周期记录能力，并且运行态存在投弹点相关状态，但这些数据尚未形成面向飞后复盘的统一消费面。用户目标是让另一台设备上的浏览器在飞后读取一次任务日志，重建飞机轨迹、计划投弹点与实际投弹点的空间关系，并支持时间轴回放。

该需求的关键约束有三点：一是本次不做实时飞机位置查看，不建设在线遥测通道；二是实际投弹点与 release 事件必须能够和轨迹日志一起被回放消费；三是用户明确希望 release 时间点日志直接作为 `flight_log` 的增量结构，而不是再引入独立事件文件。

## Goals / Non-Goals

**Goals:**
- 让 Field Editor 能加载单次飞行的历史 `flight_log`，并在地图上回放航迹与当前位置。
- 让 `flight_log` 成为飞后复盘的单一主数据源，包含 release 事件标记以及复盘所需的投弹点字段。
- 让回放视图将场地规划信息与飞后结果叠加展示，用于观察轨迹与投弹偏差。
- 保持实现边界清晰，使其与 Mission Planner 形成互补而非替代关系。

**Non-Goals:**
- 不提供实时 Web 遥测、在线飞机 marker 推送或 WebSocket/SSE 通道。
- 不复刻 Mission Planner 的 HUD、参数调节、飞控在线操作能力。
- 不在本次变更中引入新的独立飞行事件日志格式。
- 不解决视觉识别算法本身的精度问题，只消费既有投弹点结果。

## Decisions

### 1. 使用单一 `flight_log` 作为飞后回放数据源
- 决定：沿用现有 CSV flight recorder，并向其增加 release 与投弹点相关列，而不是新增并行 JSON/事件文件。
- 原因：用户已经明确要求 release 时间点日志只需作为 `flight_log` 的增量行为和结构；单文件也能避免日志关联错误。
- 备选方案：
  - 独立事件文件：结构更灵活，但需要与 `flight_log` 做时间对齐和文件绑定。
  - 运行期 API 回查：适合实时系统，不适合飞后单文件归档。

建议新增的最小字段集合：
- `release_triggered`：当前采样点是否已经观察到 release 触发。
- `release_timestamp`：首次触发 release 的记录时间戳；未触发时为空。
- `actual_drop_lat` / `actual_drop_lon`：当前已确认的实际投弹点；未确认时为空。
- `actual_drop_source`：投弹点来源，如 `vision` 或 `fallback_midpoint`。

其中 `release_timestamp` 在日志生命周期内一旦写入即保持稳定，后续行可重复该值，方便前端无需事件扫描即可快速定位投弹时刻。

### 2. Field Editor 回放视图基于离线载入，不依赖后端会话态
- 决定：Field Editor 通过用户选择历史日志文件或任务产物文件进入回放模式，在浏览器中解析并渲染轨迹。
- 原因：飞后分析天然适合离线文件驱动，且可以直接复用当前前端静态页面结构。
- 备选方案：
  - 新增后端接口列出日志目录：远端体验更完整，但会额外引入服务端 API 设计。
  - 直接访问运行目录：浏览器安全模型不支持，也不利于后续部署。

### 3. 回放 UI 复用现有地图叠加模型，新增时间轴控制层
- 决定：在 `src/field_editor/app.js` 现有 overlay 体系上增加回放轨迹线、回放位置 marker、投弹点 marker 和时间轴控制，而不是新开独立页面框架。
- 原因：现有 Field Editor 已有边界、跑道、扫描线、投弹点和任务链条的地图叠加能力，复用能降低实现成本并保持规划/复盘的一致视觉语义。
- 备选方案：
  - 独立复盘页面：结构更干净，但需要重复地图初始化和场地渲染逻辑。

### 4. 回放中的“实际投弹点”以任务确认结果为准，而非每帧视觉流
- 决定：飞后界面显示一个任务级的实际投弹点结果，并可在时间轴上高亮 release 前后轨迹；不尝试回放整个视觉检测序列。
- 原因：用户当前关心的是飞后落点与路径复盘，而不是视觉模型逐帧过程；同时现有系统也没有完整保存逐帧视觉历史。
- 备选方案：
  - 保存全部视觉时序：信息更多，但数据量、结构复杂度和价值密度都更差。

## Risks / Trade-offs

- [CSV 结构变更会影响既有分析脚本] → 在设计中要求新增列保持向后兼容，旧字段不改名，缺失新列时前端按空值处理。
- [release 事件与实际投弹点出现时间可能不同步] → 允许 `release_timestamp` 先写入，`actual_drop_*` 在后续确认后再填入后续采样行，前端分别处理“投放时刻”和“落点结果”。
- [浏览器内解析大日志时性能下降] → 首版仅要求支持单次任务常规日志规模；必要时可后续增加抽稀或分段回放。
- [规划态坐标和飞后日志坐标叠加存在 GCJ-02/WGS84 混淆] → 回放能力必须复用现有坐标转换规则，统一在前端明确日志输入坐标与地图展示坐标之间的转换路径。
- [没有实时能力可能让用户误解为功能缺失] → 在 proposal、spec 和 UI 文案中明确其定位是飞后复盘，不是实时地面站。
