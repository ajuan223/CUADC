## Context

Field Editor 当前已经具备两类能力：一类是规划态的 `field.json` 编辑、规划预览与导出；另一类是刚引入的飞后 `flight_log` 回放。现状里两套能力共享同一页面壳层、同一张地图和同一套 overlay 基础设施，但回放控件仍放在规划页的同一页面流内，容易让“编辑场地”和“复盘任务”两种工作流相互干扰。

用户已经明确要求：回放不要继续做成规划页中的对齐小卡片，而应沿用相似的整体布局语言，但作为独立 tab 进入。这意味着本次设计的关键不在于新增回放能力，而在于用最小的 UI 结构改动把两类工作流分离，同时尽量复用既有地图与渲染逻辑。

## Goals / Non-Goals

**Goals:**
- 在 Field Editor 顶部引入明确的“规划 / 回放”tab 切换。
- 保留现有双栏布局、地图容器和大部分 overlay/状态管理逻辑，避免一次性重构。
- 让规划 tab 只暴露场地编辑、预览和导出相关控件。
- 让回放 tab 只暴露 `flight_log` 导入、播放控制、回放状态摘要和复盘 overlay。
- 保证 tab 切换时地图实例可复用，避免重复初始化地图或打散既有绘图逻辑。

**Non-Goals:**
- 不重新设计 Field Editor 的整体页面框架。
- 不把规划态逻辑和回放态逻辑彻底拆成两个独立页面或路由。
- 不在这次改造中新增实时遥测能力。
- 不重构 `logic.mjs` 中与回放解析无关的规划算法。

## Decisions

### 1. 采用顶层 tab，而不是继续在单页纵向堆叠
- 决定：在 `src/field_editor/index.html` 中加入顶层 tab 导航，分别切换“规划”与“回放”两个模式。
- 原因：这最直接满足用户对“独立入口”的要求，同时保留现有页面和地图壳层。
- 备选方案：
  - 继续在同页用 panel/card 分区：实现最省，但违背用户要求，工作流边界仍然不清晰。
  - 独立 HTML 页面：边界最清楚，但会重复初始化、状态同步与样式维护成本。

### 2. 保留单地图实例，按 mode 控制侧栏和 overlay 可见性
- 决定：规划与回放共享一张地图实例；通过 app state 中的顶层 mode/tab 状态，决定哪些侧栏区块显示、哪些 overlay 在当前模式下渲染。
- 原因：地图初始化、凭据、基础 overlay 管理已经成熟，复用成本最低。
- 备选方案：
  - 每个 tab 各自维护独立地图：隔离性更强，但会重复 AMap 初始化与 overlay 管理。

### 3. 规划 tab 保持现有交互语义，回放 tab 不反向污染 field profile 表单
- 决定：规划 tab 继续使用当前 form、工具栏和 planning summary；回放 tab 只消费 `flight_log` 与回放状态，不把回放结果写回场地配置。
- 原因：规划数据与飞后结果是两套不同语义，强绑定会增加误操作风险。
- 备选方案：
  - 在回放 tab 中直接编辑规划参数：灵活但会把复盘和规划重新耦合。

### 4. 最小改造先做 DOM 分组与 mode gating，不先拆分 app.js 文件
- 决定：首版仍在 `app.js` 中实现 tab 切换和 mode gating，只把 DOM 结构拆为规划区与回放区。
- 原因：当前诉求是最小改造方案，优先控制实现面。
- 备选方案：
  - 同步把 `app.js` 大拆分成 planning/replay 子模块：长期更干净，但不属于这次最小改造目标。

## Risks / Trade-offs

- [单地图复用导致模式切换分支增多] → 用明确的 `activeTab` / `viewMode` 状态集中控制 DOM 与 overlay，可在后续再模块化。
- [已有规划态按钮和回放态按钮共存时容易出现隐藏但仍生效] → 所有事件处理需要以当前 tab 为准，隐藏区块的操作入口必须显式禁用或不渲染。
- [规划 overlay 与回放 overlay 在回放 tab 中是否同时显示存在边界歧义] → 保持现有“可对照”的设计，但以回放 tab 为主入口承载这些 overlay，而不是让规划 tab 承担回放控制。
- [最小改造后 `app.js` 继续变大] → 本次接受这个 trade-off，后续若继续扩展回放能力，再单独做模块拆分。
