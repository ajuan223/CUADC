# Part 0: 项目概述

## 项目使命

建立一套完整的固定翼无人机自动飞行控制程序，作为「搜查打击」系统中飞控自动化环节的核心。
程序运行在机载伴飞计算机（树莓派5 / NVIDIA Orin Super）上，通过 MAVLink 协议与 CUAV X7 飞控通信，
实现从起飞到投弹到降落的全自主飞行任务。

## 任务全景 — 完整的飞行生命周期

> **v2.2 更新**: 基于实际比赛场景重新设计任务流程。
> 飞行在**固定场地**内进行，跑道、围栏、扫场路线全部预先配置。
> 新增 SCAN（扫场）阶段 + 单程投弹流（视觉投弹点 / 兜底中点） + 释放机构。

# Part 2: 固定翼精化架构

## 固定翼的本质差异 — 这改变了一切

你说「仅固定翼」，这是一个非常重要的约束。固定翼和多旋翼在自动飞行控制上有根本性区别：

```
  多旋翼 (Copter)                        固定翼 (Plane)
  ═══════════════                        ═══════════════

  • 可以悬停                              • 不能悬停，必须持续飞行
  • 可以原地起降                           • 需要跑道起飞/降落
  • LOITER = 原地悬停                      • LOITER = 以某点为圆心画圈盘旋
  • GUIDED goto = 飞到点然后停              • GUIDED goto = 飞到点然后继续绕
  • 投弹可以悬停在正上方                    • 投弹必须在飞行中释放 (弹道计算)
  • LAND = 原地垂直降落                    • LAND = 需要进近航线 + 下滑道
  • 速度独立于飞行                          • 失速 = 坠毁 (有最低速度约束)
```

这意味着你的状态机和投弹逻辑必须按照固定翼的物理特性来设计。

## ArduPlane 飞行模式映射

根据我从 ArduPilot 官方文档挖到的信息，你的任务各阶段需要用到的 ArduPlane 模式：

```
  ┌────────────────────────────────────────────────────────────────────────┐
  │              ArduPlane 模式 ↔ 任务阶段映射 (v2.2 更新)                  │
  ├─────────────┬──────────────────┬───────────────────────────────────────┤
  │ 任务阶段     │ ArduPlane 模式    │ MAVLink 实现                         │
  ├─────────────┼──────────────────┼───────────────────────────────────────┤
  │ 起飞        │ AUTO             │ NAV_TAKEOFF 任务项 (需要设定爬升高度    │
  │             │ (含TAKEOFF航点)   │   和最小空速)                          │
  ├─────────────┼──────────────────┼───────────────────────────────────────┤
  │ 扫场         │ AUTO             │ 预设的 scan_pattern 航点序列           │
  │             │ (含扫场航点序列)  │ MISSION_COUNT + MISSION_ITEM_INT 上传 │
  │             │                  │ 通过 MISSION_ITEM_REACHED 监控进度     │
  │             │                  │ 最后一个航点完成 → 切换到 LOITER       │
  ├─────────────┼──────────────────┼───────────────────────────────────────┤
  │ 盘旋等待    │ LOITER           │ MAV_CMD_NAV_LOITER_UNLIM              │
  │             │                  │ 飞机以 WP_LOITER_RAD 为半径画圈        │
  │             │                  │ 保持进入盘旋时的高度                    │
  │             │                  │ 超时后可回到 SCAN 重扫                 │
  ├─────────────┼──────────────────┼───────────────────────────────────────┤
  │ 飞向目标    │ GUIDED           │ MISSION_ITEM_INT (current=2)          │
  │             │                  │ MAV_CMD_NAV_WAYPOINT                  │
  │             │                  │ MAV_CMD_DO_CHANGE_SPEED 控制巡航速度   │
  ├─────────────┼──────────────────┼───────────────────────────────────────┤
  │ 投弹进近    │ GUIDED           │ MISSION_ITEM_INT → 投弹释放点          │
  │             │                  │  (保持稳定高度和航向)                  │
  ├─────────────┼──────────────────┼───────────────────────────────────────┤
  │ 投弹释放    │ GUIDED            │ MAV_CMD_DO_SET_SERVO                  │
  │             │                  │  (通道号, PWM脉宽)                    │
  ├─────────────┼──────────────────┼───────────────────────────────────────┤
  │ 降落        │ AUTO             │ 预设 DO_LAND_START + NAV_LAND         │
  │             │ (含LAND序列)     │ 需要进近航点 + 着陆点 + 下滑道参数     │
  │             │ 或 AUTOLAND      │ (ArduPlane 4.5+ 新增 AUTOLAND 模式)   │
  ├─────────────┼──────────────────┼───────────────────────────────────────┤
  │ 人工接管    │ MANUAL/FBWA/FBWB │ RC Override 信号 (永远最高优先级)      │
  └─────────────┴──────────────────┴───────────────────────────────────────┘
```

> **关键发现**：ArduPlane 的自动降落比较复杂，需要预先在 Auto 任务里写好 `DO_LAND_START` → 进近航点 → `NAV_LAND` 的序列。或者用 ArduPlane 4.5+ 新增的 `AUTOLAND` 模式。我们的代码需要在起飞前就把降落序列写入飞控。
>
> **扫场关键**：SCAN 阶段使用 AUTO 模式执行预设航点序列。航点通过 MAVLink Mission Protocol 上传：
> `MISSION_CLEAR_ALL` → `MISSION_COUNT` → 逐条 `MISSION_ITEM_INT` → `MISSION_ACK`。
> 完成后飞控发出 `MISSION_ITEM_REACHED` 消息，Striker 检测到最后航点完成后切换 LOITER。

### 强制投弹随机点生成算法

```
  ┌─────────────────────────────────────────────────────────────────────┐
  │  ForcedStrikePointGenerator                                         │
  │                                                                     │
  │  输入: geofence (围栏多边形)                                        │
  │  输出: 一个围栏内的随机 GPS 坐标                                    │
  │                                                                     │
  │  算法:                                                              │
  │  1. 计算围栏多边形的外接矩形 (bounding box)                         │
  │  2. 在矩形内随机生成点                                              │
  │  3. 检查点是否在多边形内 (point-in-polygon, 射线法)                 │
  │  4. 不在 → 重新生成, 在 → 返回                                     │
  │  5. 排除跑道附近区域 (安全缓冲)                                     │
  │  6. 使用固定 seed 可复现 (便于调试)                                 │
  │                                                                     │
  │  安全约束:                                                          │
  │  • 随机点高度 = scan_pattern.cruise_alt_m (保持当前巡航高度)        │
  │  • 随机点距离围栏边界 ≥ 安全缓冲 (如 50m)                           │
  │  • 随机点距离跑道 ≥ 安全缓冲 (如 100m)                              │
  └─────────────────────────────────────────────────────────────────────┘
```

### 自适应频率策略

```
  视觉传入频率           程序行为
  ─────────────          ──────────────────────────────────
  0 Hz (无数据)          LOITER 状态持续盘旋等待

  单次传入 (1次)         立即采纳坐标，切换到 ENROUTE

  低频 (< 1 Hz)          每次收到都更新目标，ENROUTE中修正航线

  高频 (> 1 Hz)          固定长度滑动窗口 (collections.deque) + 中值滤波 (Median Filter) 
                          抗击高频抖动和视觉系统丢帧带来的剧烈误差反跳

  实现方式: 极速定长队列极致摒除单次跳变点保护机制
  ┌──────────────────────────────────────────────────┐
  │  class DropPointTracker:                            │
  │      ...                                         │
  │      lat_window: deque[float]  # maxlen=5        │
  │      lon_window: deque[float]  # maxlen=5        │
  │                                                  │
  │      @property                                   │
  │      def effective_target(self) -> GpsDropPoint:     │
  │          # 单次 → 直接返回                        │
```

## 固定翼投弹弹道解算 — 关键物理模型

固定翼投弹是这个项目最核心的算法挑战。因为飞机不能悬停，必须在飞行中释放，物体会沿抛物线下落：

```
  ┌─────────────────────────────────────────────────────────────────────┐
  │              FIXED-WING BALLISTIC DROP MODEL                        │
  └─────────────────────────────────────────────────────────────────────┘

  侧视图:
                               v_ground
              飞机 ═══════════════════════▶  飞行方向
               ╲ 释放点 (release point)
                ╲
                 ╲  载荷自由落体
                  ╲  + 初始水平速度 v_ground
                   ╲  + 风的影响
                    ╲
                     ╲  h (释放高度)
                      ╲
                       ╲
                        ●  目标 (target)
             ◄──────────►
                d_forward (沿飞行方向提前距离)

  基础公式:
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  下落时间:    t_fall = √(2h / g)
  
  其中 g ≈ 9.81 m/s²

  提前距离(无风): d = v_ground × t_fall

  含风补偿:
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  d_x = (v_ground_x + W_x) × t_fall
  d_y = (v_ground_y + W_y) × t_fall

  其中 W_x, W_y 是风速分量
  (ArduPilot EKF 会输出 WIND 消息，包含 wind_speed 和 wind_dir)

  考虑空气阻力 (增强模型):
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  t_fall = 更复杂，需要数值积分
  实际下落时间 > 理想自由落体 (因为空阻减慢下落)
  水平距离更短 (空阻让水平速度衰减)

  这里建议: 先用无阻力模型上线，然后通过实飞标定修正系数

  释放坐标计算 (借助 geopy 的无畸变真实极坐标推演):
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  d_total_meters = hypot(d_north, d_east)
  bearing_deg = degrees(atan2(d_east, d_north))
  release_pt = geopy.distance.distance(meters=d_total_meters).destination(target, 180 + bearing_deg)

  需要的遥测数据:
  ┌───────────────────────────────────────────────┐
  │  GLOBAL_POSITION_INT → lat, lon, relative_alt │
  │  VFR_HUD             → groundspeed, heading   │
  │  WIND                → wind_speed, wind_dir    │
  │  ATTITUDE            → roll, pitch, yaw        │
  └───────────────────────────────────────────────┘
```

# Part 3: AI 治理架构 — 三层渐进式治理

> **核心原则 (v2.1)**:
> 随着代码库增长，规范必然膨胀。将所有规范塞进一个 AGENTS.md 会导致
> AI「指令预算」溢出 (研究表明 ~150-200 条高质量指令是 LLM 有效跟踪上限)。
>
> **解决方案**: 采用 **Progressive Disclosure（渐进披露）** 模式——
> 顶层 AGENTS.md 只保留宪法级 (<100行) 规范 + Skill 路由表，
> 模块级规范作为标准 Skill 按需加载。

### Conflict 检测流程

```
  开发者/AI 修改 pkg/{name}/ 代码
       │
       ├──── 更新了 version? ────── 否 ──▶ CI lint 报错:
       │                                  "pkg/{name} source changed
       │                                   but version not bumped"
       │
       ├──── version 满足根 pyproject.toml 约束?
       │     ├── 是 ──▶ uv lock 成功 ✓
       │     └── 否 ──▶ uv lock 报 CONFLICT, 阻塞:
       │               "geodesy 0.9.0 does not satisfy >=1.2.0"
       │
       └──── REGISTRY.md 条目更新? ──── CI lint 检查 ✓
```

# Part 5: 风险与下一步

## 已知风险和待探索项

| 风险/未知 | 影响 | 建议 |
|-----------|------|------|
| ArduPlane 自动降落需要预设航线 | 降落阶段设计复杂 | 起飞前上传降落序列到飞控 |
| ~~pymavlink 与 Python 3.14 兼容性~~ | ~~已规避~~ | ✅ 改用 3.13，无兼容风险 |
| 投弹精度严重依赖风速估算 | 解算误差 | 先用简单模型 + 实飞标定修正系数 |
| RPi5 GPIO vs Orin GPIO API 不同 | 平台切换 | gpiod 统一抽象 (Linux通用) |
| 视觉链路传入格式必须为GPS绝对坐标 | 上游需自行转换 | CoordConverter 工具库供上游调用 |
| LOITER 半径受限于飞行器最小转弯半径 | 不能设太小 | config 可调, 默认 80m |
| 空速管故障 = 固定翼致命 | 安全 | 必须在 PREFLIGHT 检测 |
